# Test Creation Discipline

Default to adding no tests. Tests are production code with maintenance, CI, and debugging cost. Add one only when it meaningfully protects against a major regression or a consequential edge case.

Do not add tests merely because code changed, a new function exists, coverage can increase, or a test is easy to write. Do not create slop tests.

## Admission Gate

Add a test only when every answer is yes:

1. **Concrete risk:** Can you name the exact production behavior that could regress?
2. **Meaningful impact:** Would that regression materially affect correctness, an external contract, security, lifecycle, compatibility, data integrity, performance, or a realistic edge case?
3. **Sensitive test:** Would the test fail before the fix, under a plausible future regression, or under a targeted mutation of the protected behavior?
4. **Unique protection:** Is the behavior not already enforced adequately by the compiler, type system, generated parser, linter, build, or an existing test?
5. **Behavioral boundary:** Does the test exercise observable behavior rather than restating implementation details?
6. **Low carrying cost:** Is this the smallest deterministic test that protects the invariant without brittle setup or excessive mocking?

If any answer is no or speculative, do not add the test. State briefly why no new test is warranted.

## High-Value Risks

Tests are usually justified for:

- Reproducing a real bug that could recur.
- Protocol, serialization, CLI, API, or packaging contracts consumed outside the implementation.
- Cancellation, shutdown, resource cleanup, recovery, retry, or failure propagation.
- Concurrency, ordering, idempotency, or partial-failure invariants.
- Security boundaries and permission enforcement.
- Non-obvious normalization, validation, or error behavior where a bad input causes a material failure.
- Cross-component behavior that can compile successfully while failing at runtime.

Impact alone is not enough. The proposed test must uniquely detect the risky behavior.

## Slop Tests to Reject

Do not add tests that only:

- Assert that a newly exported symbol exists or is callable.
- Assert that an import succeeds when the build and normal use already require it.
- Pin a compatibility alias that is not required.
- Mock both sides of a one-line wrapper and assert that arguments were forwarded.
- Assert an exact mock call sequence when that sequence is not the contract.
- Exercise a standard happy-path parse already generated by a mature parser library.
- Round-trip plain constructor fields or getters with no custom behavior.
- Test a private helper introduced solely to make trivial logic testable.
- Pin comments, logging text, implementation structure, or other incidental details.
- Duplicate nearby coverage with slightly different inputs.
- Chase line, branch, or file coverage without a concrete failure mode.

Never add production helpers, aliases, hooks, or indirection merely to create a test seam for trivial behavior. Delete the proposed test instead of making the production design worse to accommodate it.

## Workflow

1. Inspect the changed behavior and existing coverage before proposing tests.
2. Write this justification in one sentence:

   `Regression: <behavior> could break when <cause>, causing <material impact>; this test catches it at <observable boundary>.`

3. If the sentence cannot be completed concretely, add no test.
4. Prefer extending an existing test over creating a new file or fixture.
5. Prefer real behavior at the narrowest meaningful boundary over mocks. Use an integration test only when the integration boundary is the risk.
6. Protect one invariant with one test. Use parameterization only when each case represents a distinct boundary of that same invariant.
7. When practical, verify sensitivity by running the test against the pre-fix behavior or a targeted mutation, then run the smallest relevant test command.

An explicit request to "add tests" does not waive this gate. Add the minimum qualifying set and omit the rest.

## PR 11636 Examples

Use these examples from `ai-dynamo/dynamo#11636` as calibration:

- Reject `assert callable(backend._run_sglang_sidecar)`. It only restates that the binding exported a symbol and does not execute the launcher contract.
- Reject tests that replace both `sidecar.main` dependencies with mocks and assert argument forwarding or default `sys.argv` handling. The wrapper was direct, and the tests only mirrored its implementation.
- Reject a Clap test that parses one ordinary `--sglang-endpoint` invocation. Derive-generated happy-path parsing was not a meaningful project invariant.
- Reject a test for a helper that merely prepends the executable name to a vector. The helper itself was unnecessary production indirection created around trivial behavior.
- Remove an import-side-effect test once the import-time logging action has been deleted. With no side effect left, the test protects absence of code rather than a fragile mechanism.
- Keep focused endpoint-normalization coverage. Rejecting empty hosts and unsupported TLS schemes while translating bare and `grpc://` endpoints protects non-trivial connection behavior and consequential edge cases.
- A real process-level executable test can be justified when the executable name and CLI are an active external contract. Do not create such a test for a future or explicitly out-of-scope contract.

## Reviewing Existing Proposals

Judge each proposed test independently. Do not keep a weak test because it is already written or because deleting it reduces coverage. Recommend removal when it fails the admission gate.

Do not use this skill as blanket authority to delete unrelated existing tests. Preserve tests that still protect meaningful behavior, and remove existing tests only when requested, obsolete, duplicative, or clearly slop within the active change.

## Handoff

For each new test, report the exact regression or edge case it protects. When adding none, say so plainly and name the existing validation or low-risk reason; zero new tests is often the correct result.
